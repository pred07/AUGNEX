## 2. Core Content

### The Problem with Physical Hardware
In the old days, one server ran one application.
*   **Inefficient**: Most computers sit idle 90% of the time.
*   **Dangerous**: If you run malware on your main PC to analyze it, you risk infecting your personal files.

### The Solution: Virtualization
Virtualization inserts a software layer (the **Hypervisor**) between the hardware and the operating system.
*   **Abstraction**: It tricks the operating system into thinking it has its own dedicated CPU, RAM, and Disk.
*   **The Capsule**: Each VM is a self-contained file. You can copy it, delete it, or "snapshot" it (save its state) instantly.

### Security & Isolation
This is the most critical concept for us.
A VM is a **Sandbox**.
*   **Malware Analysis**: You can detonate a virus inside a VM. If it destroys the VM, your physical host remains untouched (usually).
*   **Attack Simulation**: You can attack a VM without getting arrested for attacking real internet infrastructure.

## 3. Guided Practice

### Checking for Virtualization Support
Before we can run VMs, your CPU must support hardware virtualization (VT-x for Intel, AMD-V for AMD).

**Windows Users:**
1.  Right-click the **Taskbar** and select **Task Manager**.
2.  Click the **Performance** tab (the squiggly line icon).
3.  Click **CPU**.
4.  Look for "Virtualization: **Enabled**" in the bottom right info block.

**Linux Users:**
Run this command:
```bash
lscpu | grep Virtualization
```
*   Expected Output: `VT-x` or `AMD-V`.

**Troubleshooting**:
If it says **Disabled**, you will need to reboot into your BIOS/UEFI settings and enable "Intel VT-x" or "SVM Mode".
