# Validation Without Exploitation

## 1. Orientation

### What this module covers
This module covers checking **Configuration** instead of simulating **Attacks**. "Do we have the lock?" vs "Can we pick the lock?"

### Fit in the Learning Path
You don't need to burn the house down to test the smoke detector.

### Learning Objectives
By the end of this module, you should understand:
*   **Static Configuration Analysis**.
*   **Compliance Scanning**.
*   **Assumption Validation**.

---

## 2. Core Content

### Config vs Exploit
*   **Exploit approach**: Try to run Mimikatz to see if LSA Protection is on.
*   **Config approach**: Query Registry `HKLM\SYSTEM\CurrentControlSet\Control\Lsa` for `RunAsPPL`.
    *   If `1`, we are safe.
    *   *Benefit*: Instant, safe, 100% coverage.

### Tools
*   **ScoutSuite** (Cloud Config).
*   **OpenSCAP** (Linux Hardening).

---

## 3. Guided Practice

### Checking S3 Security
**Exploit Way**: Try to `curl` every bucket URL to see if it lists files. (Noisy, slow).
**Config Way**: Query AWS API `GetBucketPolicy`. (Fast, silent).
*   If `Principal: *`, alert.

### Checking Patch Levels
**Exploit Way**: Throw `EternalBlue` at the server. (DANGEROUS).
**Config Way**: Check `systeminfo` for KB4012212. (Safe).

---

## 4. Reflection Check

1.  **Limitations**: Does config prove security? (No. The registry might say "On", but a kernel bug might bypass it. Exploitation is the *ultimate* proof. Config is a proxy).
2.  **Combo**: Use Config for scale (10,000 hosts). Use Exploit for depth (Critical hosts).
3.  **Speed**: Config checking is instant.

---

## 5. Completion Criteria

This module is complete when:
1.  You prioritize **Querying State** over **Throwing Exploits**.
2.  You use tools like **ScoutSuite**.
3.  You understand the limitations of "Paper Safety".
